# MaintOps Web Console

Spanish documentation: [README.es.md](README.es.md).

MaintOps Web Console is the Vue 3 browser application for the MaintOps vehicle maintenance operations platform. It gives users a role-aware operational console for dashboard review, customer and vehicle records, workshops, maintenance catalogs, orders, audit logs, realtime activity, and analytics.

This project is designed as the frontend piece of the full MaintOps portfolio stack. It can run by itself for UI review, but the complete experience depends on:

- `maintops-api-laravel` for authentication, authorization, business data, password recovery, and service-token issuing.
- `maintops-realtime-node` for live operational events and presence.
- `maintops-analytics-fastapi` for metrics, forecasts, risk alerts, and recommendations.
- `maintops-stack` for the reproducible local environment that runs all services together.

## What This Project Demonstrates

- Vue 3 application architecture with route guards, shared stores, feature modules, and reusable UI primitives.
- A professional admin console built from a proven UI foundation and adapted to a real maintenance domain.
- Docker-first frontend execution without requiring Node.js on the host.
- Role-aware navigation and action visibility that mirrors backend policy intent without replacing backend authorization.
- Authenticated Laravel API integration with normalized error handling and pagination/query helpers.
- Public password recovery screens that consume Laravel auth endpoints and support reset links generated by email.
- Authenticated Socket.IO lifecycle using short-lived Laravel-issued realtime tokens.
- Realtime notification center that reacts to operational events without letting the browser manage rooms.
- Analytics integration using dedicated Laravel-issued analytics tokens instead of the main API token.
- English product interface with parallel English and Spanish technical documentation.

## MaintOps Ecosystem Role

The frontend is intentionally a presentation and integration layer. It improves user experience by hiding unavailable actions, grouping workflows, and refreshing data after realtime events, but Laravel remains the source of truth for authentication, authorization, validation, and business decisions.

The browser communicates with three external application services:

```text
Laravel API      VITE_API_BASE_URL
Realtime Gateway VITE_REALTIME_URL
Analytics API    VITE_ANALYTICS_BASE_URL
```

The normal review path is to run this project through `maintops-stack`, where those services are already wired together.

## Project Structure

The frontend uses a compact feature-module structure:

```text
src/
  api/          Axios client, error normalization, query helpers, and API exports.
  auth/         Frontend permission matrices and UX helpers.
  components/   Template-derived reusable UI primitives.
  components/ui/ MaintOps-specific page, table, badge, dropdown, and empty-state components.
  config/       Runtime integration URLs from Vite environment variables.
  css/          Tailwind entrypoint and style variants.
  layouts/      Authenticated and guest layout shells.
  modules/      Feature modules grouped by domain capability.
  router/       Route definitions, guards, and metadata.
  stores/       Shared Pinia state.
  types/        JSDoc contracts for API and domain payloads.
  views/        Generic or top-level page views.
```

Feature modules follow the same responsibility split:

```text
modules/<feature>/
  services/     HTTP clients or integration services for that feature.
  views/        Route-level screens.
  components/   Feature-specific reusable components when needed.
  composables/   Feature-specific reactive helpers when needed.
```

This keeps shared UI primitives separate from domain-specific workflows, and keeps API access out of route components.

## Architecture Decisions

The frontend mirrors backend authorization intent but does not enforce security. Route visibility, menu items, buttons, and status actions are filtered in the browser to reduce user friction; Laravel still validates every request with policies and rules.

The API client centralizes Bearer token injection, Laravel validation error normalization, network failure messages, and query serialization. Feature modules consume that client through small services rather than calling Axios directly from every view.

Realtime events are treated as backend-owned facts. The frontend deduplicates by event id and refetches affected dashboard, list, or detail screens. This avoids maintaining a second copy of Laravel's state-machine logic in the browser.

Analytics is intentionally separated from the main API client. The browser asks Laravel for an analytics service token, then calls FastAPI with that token. The main Laravel token is not sent directly to Analytics.

Admin One provides the visual foundation, but template demo pages, sample data, promotional links, and template-specific branding are removed. MaintOps keeps the shell patterns and defines its own routes, modules, domain language, and integration model.

## Environment Variables

Create a local `.env` from `.env.example` only when you need to override service URLs or allowed hosts:

```dotenv
FRONTEND_ALLOWED_HOSTS=localhost,127.0.0.1
VITE_API_BASE_URL=http://localhost:8000/api/v1
VITE_REALTIME_URL=http://localhost:3000
VITE_ANALYTICS_BASE_URL=http://localhost:8001
```

For the full local demo, these values are provided by `maintops-stack`.

## Run With Docker

Standalone frontend review:

```bash
docker compose up -d --build
docker compose logs -f frontend
```

Open:

```text
http://localhost:5173
```

Stop:

```bash
docker compose down
```

The frontend will render by itself, but authenticated workflows require the Laravel API. Realtime and Analytics screens require their corresponding services.

## Demo Review Notes

- Use the full `maintops-stack` repository to review the integrated product.
- Use sample data only. Demo environments should not receive real personal, customer, workshop, or vehicle data.
- Laravel remains the authorization source of truth; frontend role checks only improve navigation and action visibility.
- Connectivity failures are normalized into clear API availability messages so reviewers can distinguish a service outage from an application validation error.
